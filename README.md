# Provenance Guard

Provenance Guard is a Flask-based backend system designed to evaluate text submissions and classify them based on the likelihood that they were generated by AI. It uses a multi-signal detection approach, assigns a confidence score, provides transparency labels, and includes a built-in workflow for creators to appeal classifications.

## Architecture Overview
When a payload is submitted to the system, it follows this path:
1. **Access & Protection:** A request hits the `POST /submit` endpoint and passes through a Flask-Limiter check to ensure it hasn't exceeded the rate limit.
2. **Identification:** The system mints a unique UUID (`content_id`) for the submission.
3. **Evaluation:** The text is passed to two distinct detection signals: an LLM API (Groq) and a local stylometric heuristic engine.
4. **Scoring:** The outputs from both signals are fed into a weighted scoring engine to produce a final confidence score (0.0 to 1.0).
5. **Labeling:** The confidence score determines the final transparency label (e.g., "Generated by AI").
6. **Persistence:** A structured JSON record of the classification is appended to an internal `log.json` file.
7. **Response:** The system returns a `201 Created` JSON response containing the `content_id`, confidence score, and transparency label.

## Detection Signals
The system relies on two distinct signals to evaluate text:

**Signal 1: LLM Stylistic Analysis (Groq)**
* **What it measures:** We use the `llama-3.3-70b-versatile` model (set to temperature 0.0 for determinism) to evaluate the text's overall flow, word choice, and sentence rhythm. 
* **Why this was chosen:** Large language models are highly capable of recognizing the specific semantic "flavor" and default pacing of other AI models.
* **What it misses:** It can sometimes misclassify highly stylized human text or text that heavily relies on structured, formulaic templates.

**Signal 2: Stylometric Heuristics (Python Engine)**
* **What it measures:** A pure Python function that calculates sentence-length variance (coefficient of variation) and punctuation density. 
* **Why this was chosen:** AI models tend to produce highly uniform sentence lengths and predictable punctuation patterns. Human writing is generally more chaotic and "bursty."
* **What it misses:** It struggles with intentionally rigid human writing, like legal disclaimers or step-by-step technical instructions.

## Confidence Scoring
Instead of relying on a single point of failure, the system combines both signals into a weighted score to represent our overall confidence that the text is AI-generated:

**Formula:** `(Groq Score * 0.6) + (Heuristic Score * 0.4)`

**Validation Examples:**
To ensure the scoring engine produces meaningful variation, it was tested against known inputs:
* **High-Confidence AI (Corporate Jargon):** 
  * Groq Score: 0.95
  * Heuristic Score: 0.583
  * **Combined Score: 0.803**
* **Low-Confidence (Varied Human Text):** 
  * **Combined Score: 0.008**

## Transparency Labels
The final combined confidence score determines the exact transparency label returned to the user:

* **Score >= 0.75:** `Generated by AI`
* **Score >= 0.35 and < 0.75:** `Created with AI assistance`
* **Score < 0.35:** `Human original`

## Rate Limiting
The `POST /submit` endpoint is protected by `Flask-Limiter` using an in-memory storage engine. 
* **Limit:** 5 requests per minute.
* **Reasoning:** This strict limit is specifically configured for development and testing to easily verify the `429 Too Many Requests` behavior without needing to spam the server thousands of times. It effectively prevents automated scraping and API abuse. In a production environment, this would be swapped to a Redis backend.

## Appeals Workflow
If a creator disagrees with a classification, they can submit a request to `POST /appeal` with their `content_id` and reasoning. The system searches `log.json`, flips the original log entry's status from `"classified"` to `"under review"`, and safely appends their reasoning and a timestamp to the record.

## Known Limitations
The system will likely misclassify **technical documentation and code comments** as "Generated by AI" or "Created with AI assistance." Because technical writing inherently uses rigid, short sentence structures and sparse, functional punctuation, it naturally trips the uniformity thresholds of our Stylometric Heuristic signal, falsely dragging the AI confidence score upward.

## Spec Reflection
* **How the spec helped:** The specification required using placeholders for labels and scores in earlier milestones. This isolated the complexity, allowing me to build a stable ingest and logging pipeline before introducing the math and weighting logic later on.
* **How implementation diverged:** The spec suggested potentially using a local database like SQLite for the audit log. I actively chose to diverge from this and use a lightweight, append-only `log.json` file instead. This removed the overhead of schema migrations and made visual inspection and diffing much faster during testing.

## AI Usage
AI tools (Claude/Gemini) were used during development in the following specific ways:
1. **Initial Flask Skeleton:** I prompted the AI to generate the initial `app.py` structure and the `evaluate_groq_signal` function. When the AI provided the standard Groq API call, I revised the regex parsing logic to ensure it returned a `-1.0` sentinel value upon failure rather than crashing or returning a false zero.
2. **Appeals Endpoint:** I directed the AI to write the `POST /appeal` endpoint based on my specific JSON body requirements. The AI originally wrote new inline logic to open and overwrite the log file; I overrode this by refactoring it to reuse a unified `write_log` helper I established for cleaner file handling.
